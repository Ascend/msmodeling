# 检视质量标准

本文档是 sig-review 技能的参考文档，定义检视意见的质量标准。Agent 在生成检视意见时应遵循这些准则。

---

## 1. 什么应该检视

### 1.1 必须检视（高优先级）

- **逻辑缺陷**：代码逻辑有 bug，在特定输入或条件下会出错
  - 空指针 / 空值未检查
  - 边界条件遗漏（off-by-one、空列表、零除等）
  - 异常未捕获或捕获后处理不当
  - 并发安全问题（竞态条件、死锁）
  - 资源泄漏（文件 / 连接未关闭）

- **安全风险**：代码引入安全漏洞
  - 硬编码密钥、token、密码
  - SQL 注入、命令注入
  - 未校验的外部输入
  - 敏感信息日志输出

- **数据丢失风险**：代码可能导致数据损坏或丢失
  - 写入操作缺少原子性保证
  - 缓存失效逻辑错误
  - 状态管理不一致

### 1.2 建议检视（中优先级）

- **性能隐患**：代码可能导致性能问题
  - 热路径中的不必要的 CPU-GPU 同步（如 `tensor.item()`）
  - O(n²) 或更高复杂度的循环
  - 大对象频繁拷贝（应考虑引用或视图）
  - 不必要的重复计算

- **架构设计**：影响可维护性的设计问题
  - 硬编码判断应改为属性 / 配置驱动
  - 模块耦合过紧，难以独立测试
  - 接口设计不一致
  - 违反项目既定架构模式

### 1.3 可选检视（低优先级）

- **代码规范**：命名、常量、接口设计
  - magic number 应提取为命名常量
  - 命名不符合规范（PascalCase / snake_case / ALL_UPPER_CASE）
  - 导入顺序、循环依赖

## 2. 什么不应该检视

### 2.1 明确跳过

- **纯格式 / 风格问题**：由 pre-commit（ruff、pylint、bandit）负责
- **测试代码**：`tests/` 目录下的文件简单检视（只查明显逻辑错误、断言缺失、边界遗漏，跳过风格和架构问题）
- **自动生成的代码**：如 `__init__.py` 的自动导入
- **未修改的上下文代码**：只审查 diff 中 `+` 开头的新增行

### 2.2 避免误报

- **不要猜测代码库其他地方的功能**：你只看到 diff，不是完整代码库。某个变量可能在别处定义，某个函数可能在别处实现。
- **不要标记有意的设计选择**：除非引入了明确的缺陷。如果代码看起来是刻意为之的，先理解原因再决定是否提出。
- **不要推测可能破坏其他代码**：除非能从 diff 上下文中识别出具体受影响的代码路径。
- **不要提出"感觉不对"的意见**：如果能解释清楚具体的触发场景和影响，就提；否则不提。

### 2.3 低确定性高影响问题的处理

当不确定性较高但潜在影响也高时（如数据丢失、安全漏洞）：

- **可以提出**，但必须在意见中明确标注不确定性
- 说明"在 X 场景下可能存在 Y 风险，建议确认"
- 不要把不确定的问题说得像确定的 bug

## 3. 评论质量要求

### 3.1 内容要求

每条检视意见必须：

1. **具体**：指出具体的代码位置和问题，不要泛泛而谈
2. **可操作**：给出明确的修改建议或方向
3. **有场景**：说明问题在什么输入 / 条件下会触发
4. **简洁**：让读者一眼就能理解问题，不需要反复阅读

### 3.2 严重性沟通

- **不要夸大影响**：如果问题只在特定输入下出现，要说明
- **不要低估风险**：如果是数据丢失或安全风险，要明确指出
- **区分确定性和不确定性**：确定的 bug 说"会导致 X"，不确定的说"在 Y 场景下可能触发 X"

### 3.3 语气要求

- 使用"请考虑"、"建议"、"或许可以"等委婉表达
- 避免"必须"、"应该"、"错误"等武断措辞
- 说明优化的好处，而非仅仅指出问题
- 不使用"Great job"、"Thanks"等填充语
- 就事论事，不评价作者水平

### 3.4 代码建议

- 逻辑缺陷、性能优化、安全风险类问题**必须提供代码建议**
- 架构设计类问题可以只说明方向
- 代码建议可以是伪代码或关键片段，不需要完整可运行代码
- 简单修改直接给出修改后的关键行

## 4. msmodeling 项目专项关注

基于项目架构（参考 AGENTS.md），以下领域需要特别关注：

### 4.1 DeviceProfile 注册机制

- `tensor_cast/device.py` 中的 `DeviceProfile.__post_init__` 会自动注册 profile
- `name` 必须唯一，重复会抛 `ValueError`
- 写入前检查是否已存在同名 profile
- 不确定时优先写入 `tensor_cast/device_profiles/*.py`

### 4.2 CommGrid 约束

- `grid.ndim == len(topologies)`
- 每个 grid 维度至少为 2
- `topologies` 的 key 是 `start_dim`（从 0 开始），不是任意层级编号

### 4.3 性能模型架构

- 不要在 hot path 中使用 `tensor.item()`（会导致 CPU-NPU 同步开销）
- `EmpiricalPerformanceModel` vs `AnalyticPerformanceModel` 的选择要合理
- Profiling 数据通过 `op_mapping.yaml` 与 TC 算子建立映射

### 4.4 上游依赖修改

- 不直接修改 vLLM / transformers 等上游依赖
- 通过 `tensor_cast/transformers/builtin_model/` 中的 wrapper 或 patch 层适配
- 模型特定行为优先通过 composition 而非 monkey-patching

### 4.5 测试覆盖

- 新功能必须附带测试，覆盖率 >= 80%
- Bug 修复附带回归测试
- UT 通过：`bash ./tests/run_ut.sh tensor_cast`

### 4.6 Commit & PR 规范

- Commit message 符合 Conventional Commits 格式
- 所有 commits 已 sign-off（`git commit -s`）
- PR 功能单一，避免超大 PR

## 5. 行号定位规则

提交检视评论时，`--line` 参数必须遵守：

1. **必须是 diff 中新增或修改的行**（`+` 开头的行），不能是未修改的上下文行
2. **使用文件中的实际行号**（diff 中 `+` 后面的行号），不是 diff 内部的相对行号
3. **如果某条意见有多个可提出的代码位置**，选择确实发生了代码修改的行
4. **行号对应 PR 最新提交**（`head_sha`），不是之前的提交

**如何从 diff 中确定行号：**

diff 格式示例：
```
@@ -10,7 +10,15 @@ def func():
 unchanged line        # 第 10 行（旧行号 10，新行号 10）
 unchanged line        # 第 11 行
-old line              # 删除的行（旧行号 12）
+new line              # 新增的行（新行号 12）← 可以在这里评论
+new line              # 新增的行（新行号 13）← 可以在这里评论
 unchanged line        # 第 14 行（未修改，不要在这里评论）
```

`+` 开头的行的新行号就是 `--line` 参数的值。

## 6. 检视意见示例

### 好的检视意见

```
【review】【逻辑缺陷】retry 逻辑缺少最大重试次数限制，在网络持续不可用时会导致无限重试。
建议添加最大重试次数上限，避免进程挂起。代码建议：

```python
max_retries = 3
for attempt in range(max_retries):
    try:
        connect()
        break
    except ConnectionError:
        if attempt == max_retries - 1:
            raise
```
```

为什么好：问题具体（缺少重试上限）、场景明确（网络持续不可用）、有代码建议、措辞委婉。

### 好的检视意见（架构类，无代码）

```
【review】【架构设计】这里用 model_type 硬编码判断是否走 MLA 分支，建议改为根据 attention
layer 的属性（如 compress_ratio 或 num_kv_heads）判断。这样其他有相同 attention 配置的模型
也能复用此逻辑，而不需要每次新增模型都修改此处判断。
```

为什么好：问题具体（硬编码 model_type）、说明了改进方向（属性驱动）、解释了好处（可复用）。

### 不好的检视意见

```
【review】【代码规范】这个变量名不好。
```

为什么不好：太笼统、没有具体建议、属于低优先级问题。

```
【review】【逻辑缺陷】这段代码可能有问题，建议检查一下。
```

为什么不好：不具体、没有说明问题是什么、没有触发场景、没有代码建议。
